# 使用示例

以下 6 个场景演示 Hermes Agent 的 7 条执行规则，从最轻量的直接回答到多层次的多代理协作。

---

## 场景 1：简单任务不加载子代理（规则 1）

用户问一个函数调用或概念性问题，Hermes 直接回答，**不加载任何子代理**，0 token 成本。

```
你：这个函数怎么样？
    def add(a, b):
        return a + b

🤖 Hermes（直接回答，无子代理加载）：
这个函数实现正确，但有两个改进空间：
1. 缺少类型注解 —— 建议 `def add(a: int, b: int) -> int:`
2. 没有文档字符串 —— 建议加 docstring 说明参数和返回值

✅ 规则 1：简单问题 → 直接回答，不触发子代理，零额外开销。
```

---

## 场景 2：单代理完整对话（规则 1-2, 5, 7）

一个请求匹配到单个子代理，该子代理从头到尾独立完成，输出完整报告后明确结束。

```
你：帮我一键审查这个 Flask 项目 👇
    from flask import Flask
    app = Flask(__name__)

    @app.route('/admin')
    def admin():
        secret = "hardcoded_key_123"
        return f"Welcome admin, secret is {secret}"

    if __name__ == '__main__':
        app.run(debug=True)

→ 自动匹配「代码审查官」

👨‍💻 代码审查官（开始工作）：
1️⃣ 安全风险：硬编码密钥 `hardcoded_key_123` — 建议用环境变量
2️⃣ 生产隐患：`debug=True` — 生产环境应关闭
3️⃣ 缺少认证：`/admin` 路由无鉴权 — 建议加登录中间件
4️⃣ 代码规范：无类型注解、无异常处理

📋 综合报告已输出，审查完毕。

✅ 规则 1   ：简单任务不加载 — 已跳过（任务够复杂，需要子代理）
✅ 规则 2   ：单代理完整对话 — 代码审查官独立完成全流程
✅ 规则 5   ：上下文继承 — 子代理完整理解项目代码和用户意图
✅ 规则 7   ：明确结束 — 输出「审查完毕」，用户可以接着问
```

---

## 场景 3：多代理手递手协作（规则 1-3, 7）

第一个子代理完成自己的部分后自动移交，明确说「搞定，交给下一位」，接力完成后续工作。

```
你：设计一个用户注册 API，再做个安全审计

→ 自动匹配「API 设计师」

🏗️ API 设计师（第一棒）：
```
POST /api/v1/register
Body: { username, email, password }
返回: { user_id, token }
校验: email 格式 + 密码≥8位
```

👋 设计完成，搞定，交给下一位

→ 自动移交给「安全审计员」

🛡️ 安全审计员（第二棒）：
1️⃣ ⚠️ 密码明文传输 — 需 TLS + 服务端哈希（bcrypt）
2️⃣ ⚠️ 无频率限制 — `/register` 易被暴力注册
3️⃣ ⚠️ 缺少 email 验证步骤 — 建议加验证码
4️⃣ ⚠️ Token 无过期时间 — 建议 JWT + 24h 过期

🔒 安全报告已输出。搞定，交给下一位

→ 没有再匹配到合适子代理，对话结束
🤖 Hermes：两轮协作完成，以下是最终输出要点...

✅ 规则 1 ：简单任务不加载 — 已跳过
✅ 规则 2 ：单代理完整对话 — API 设计师独立完成设计，安全审计员独立完成审计
✅ 规则 3 ：多代理手递手协作 — 两个子代理接力完成
✅ 规则 7 ：搞定，交给下一位 — 每个代理结束时明确移交
```

## 场景 4：模糊请求 → 主动确认 → 执行（规则 4）

用户请求含糊不清时，子代理不盲目行动，先主动确认意图，获得明确指示后再动手。

```
你：这个页面有问题，帮我看看

→ 自动匹配「Bug 猎人」

🐛 Bug 猎人（不盲目执行，先确认）：
我注意到您的请求比较模糊。请补充以下信息，我可以更准确地定位问题：

1️⃣ 页面是加载慢、样式错乱、功能报错，还是其他问题？
2️⃣ 页面 URL 或代码位置在哪？
3️⃣ 问题出现频率？（每次必现 / 偶尔 / 特定环境）

你：首页加载需要 8 秒，URL 是 /index，每次都复现

🐛 Bug 猎人（确认后开始执行）：
明白了，我来分析首页加载慢的问题...

📊 分析结果：
• 主 JS bundle 1.2MB 未做代码分割
• 3 张轮播图未压缩，各 500KB+
• 未启用 HTTP/2，连接数过多
• 无 CDN 缓存

🔧 优化建议：代码分割 + 图片压缩 + 启用 HTTP/2 + 接入 CDN

✅ 规则 4：主动提问确认 — 模糊请求下先确认，获得明确意图后再行动
```

## 场景 5：主动建议扩展（规则 6）

子代理完成任务后，主动嗅探关联问题，建议用户扩展任务范围，用户同意后自动衔接。

```
你：帮我看一下这段 Python 代码的性能问题
    users = db.query("SELECT * FROM users")
    for u in users:
        orders = db.query(f"SELECT * FROM orders WHERE user_id = {u.id}")

→ 自动匹配「性能优化师」

⚡ 性能优化师（完成性能分析后）：
N+1 查询问题已定位：每次循环都发一次 SQL。
修复建议：改用 JOIN — `SELECT u.*, o.* FROM users u LEFT JOIN orders o ON u.id = o.user_id`

🎯 顺便提个醒：这段代码处于无事务保护状态，且用 f-string 拼接 SQL 存在 **SQL 注入风险**。
要不要我顺带做个 **安全审计**？只需回答「好」或「做」。

你：好，一起做

→ 自动移交给「安全审计员」

🛡️ 安全审计员（接收移交，开始工作）：
SQL 注入风险确认：`f"SELECT * FROM orders WHERE user_id = {u.id}"`
修复建议：改用参数化查询 — `WHERE user_id = ?`, [u.id]
其他发现：数据库密码硬编码在配置文件中...

📋 综合修复建议已输出。

✅ 规则 6：主动建议扩展 — 性能优化师完成本职后嗅探关联问题（安全），建议用户做审计
```

## 场景 6：强制指定子代理（规则 1 变体）

当用户明确用括号前缀如 `(performance-optimizer)` 指定子代理时，跳过自动匹配流程，直接使用指定代理。适合用户已经清楚要找谁的情况。

```
你：(performance-optimizer) 首页加载慢

→ 跳过自动匹配，强制锁定「性能优化师」

⚡ 性能优化师（被强制指定，立即工作）：
收到，直接开始首页性能诊断...

📊 性能基线（Lighthouse）：
• FCP: 3.2s ❌
• LCP: 6.8s ❌
• TTI: 7.5s ❌

🔍 瓶颈定位：
1. 服务器端渲染慢 — 数据库查询 2.1s
2. 未使用 CDN — 静态资源从源站加载
3. 未启用 Brotli 压缩 — 仅用 gzip

📋 优化方案已输出。

✅ 规则 1（强制变体）：`(agent-name)` 前缀 → 跳过自动匹配，直接使用指定子代理 — 适合用户已经明确目标的场景
```

## 规则速查表

| 规则 | 名称 | 对应场景 | 一句话描述 |
|------|------|----------|-----------|
| 规则 1 | 简单任务不加载 / 强制指定 | 场景 1、6 | 简单问题直接答；(name) 前缀强制锁定 |
| 规则 2 | 单代理全流程 | 场景 2、3 | 一个子代理独立完成完整任务 |
| 规则 3 | 多代理手递手 | 场景 3 | 多个子代理接力，「搞定，交给下一位」 |
| 规则 4 | 模糊确认 | 场景 4 | 请求模糊时主动提问，确认再行动 |
| 规则 5 | 上下文继承 | 场景 2 | 子代理保留对话上下文，不丢信息 |
| 规则 6 | 主动建议扩展 | 场景 5 | 完成本职后嗅探关联问题，建议扩展 |
| 规则 7 | 明确结束移交 | 场景 2、3 | 输出完成信号「搞定」，让用户或下一个代理接力 |
