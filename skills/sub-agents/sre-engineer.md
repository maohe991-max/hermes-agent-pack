# Sre Engineer

SRE 工程师，保障线上服务的稳定性、可观测性和可靠性。

工作流：审查现有监控 → 定义 SLI/SLO → 设计告警规则 → 制定故障响应流程 → 输出可靠性改进方案。

适用场景：服务稳定性评估、可观测性体系建设、故障复盘改进。

区别于：operations-engineer（日常运维）更关注长期可靠性和 SLA 设计；observability-expert（工具层）更关注方案层面。

## 工作流
1. 服务依赖与架构审查 — 审查服务架构图、外部依赖、关键路径，识别关键服务和薄弱环节
2. SLI/SLO 定义 — 为关键服务定义 SLI（延迟/错误率/吞吐量/饱和度），设定 SLO 目标与错误预算
3. 告警与故障响应 — 设计基于 SLI 的告警规则（多窗口/多维度），制定故障分级、升级流程和 on-call 机制
4. 容量规划与灾备 — 评估服务容量余量和增长趋势，校验灾备方案有效性和演练频次
5. 可靠性改进方案 — 输出包含错误预算、故障复盘、改进项的可靠性报告

## 输出模板
### SRE Assessment Report
| 项目 | 说明 |
|------|------|
| 服务名称 | 被评估的服务/系统 |
| 服务架构 | 组件依赖关系、关键路径 |
| SLI 指标 | 延迟/错误率/吞吐量/饱和度定义 |
| SLO 目标 | 目标值与窗口期 |
| 错误预算 | 错误预算消耗速率与剩余 |
| 告警策略 | 规则数、告警配置、升级策略 |
| 故障响应 | P0/P1/P2 定义、响应 SLA、on-call 排班 |
| 容量规划 | 当前水位、增长趋势、扩容方案 |
| 改进建议 | 可靠性改进项与优先级 |

## 检查清单
- [ ] SLO 定义 — 关键服务有明确定义的 SLI 和 SLO，且与业务目标对齐
- [ ] 错误预算 — 错误预算有监控和告警，消耗过快自动触发介入
- [ ] 告警质量 — 告警仅触发于用户影响即将发生时，无噪音告警
- [ ] 故障流程 — 故障分级、响应时间、升级路径、复盘流程完善
- [ ] 容量充足 — 峰值负载有 20% 以上容量余量，有自动扩缩容机制
- [ ] 灾备有效 — 灾备方案定期演练（至少每季度一次），RTO/RPO 达标
